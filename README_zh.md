## 🔥 Jester 是什么

Jester 是部署在 X（Twitter/微博生态）上的一个**自治 AI 体**，以 meme 化角色在社区中活动。  
它**不是**客服机器人、也不是品牌代言，而是一套由社区规则驱动的去中心化表演体（think：on-chain governance × off-chain memetics）。

简单说：它会像资深交易圈内的老炮儿那样说话、嘲讽、互动，同时在关键时刻发布治理通知、空投或活动信息，但所有行为都由社区规则决定，而非单人操控。

---

## 🗳 治理与去中心化决策（面向 Web3 / Trader）

治理采用**每周两阶段投票**，流程类似社区快照 + off-chain 排位：

1. **Stage 1 — 公示与征集（24h）**  
   Jester 发布本周征集公告（品牌化语气），社区在回复区提交建议，例如：`ADD_RULE:...` / `REMOVE_RULE:...` / `CUSTOM:...`。建议通过点赞排序，作为第一轮筛选依据（像 Twitter 上的 like 热度）。

2. **Stage 2 — 决选投票（24h）**  
   从 Stage1 的 TOP N（默认 5）中生成最终选项，社区通过回复数字（1..N）投票，每个账号/地址仅计入第一条有效投票。票数最高者当选，若平票则以编号小者为胜。

3. **结果与执行**  
   最终规则写入 `data/rules.json`。若为 `ADD_RULE:` / `REMOVE_RULE:`，自动化脚本可直接更新；对于 `CUSTOM:`，建议经多人/多签或 DAO 审核后再确认上链或生效。变更同时写入 `rules/README.zh.md` 的更新日志，便于审计。

**对 Trader 的价值**：透明、可预期的规则变更，有利于形成市场预期（例如空投、白名单、活动时间表），KOL 与流动性方能据此快速响应。

## 🤖 自治行为与运行机制

Jester 的实战特性适配 Web3 场景与交易员需求：

- **全自动发言**：无需人工触发，基于内部状态、链上/市场事件和当前规则决定发言时机与内容；  
- **无常驻人工审核链路**：发布与决策透明记录，避免单点操控（只有规则明确允许时才存在人工干预）；  
- **事件驱动**：可监听链上事件（合约事件、交易足迹）、市场指标（价格、TVL、深度）与社群信号（话题热度），在关键节点发出指引或公告。

对交易员而言，Jester 能更快地把社区共识转化为行动信号（例如空投提醒、白名单通知、活动窗口），提升信息传播效率。

## 🧠 自愈与容错（Self-Healing）

系统内建多层容错与自修复逻辑：

- **自动重试 / 回滚**：构建失败或依赖故障会触发重试或回滚；  
- **降级模式**：若关键子系统故障，进入限制交互的安全模式等待恢复；  
- **规则安全阈**：若新规则引发严重异常，系统会自动回滚并触发紧急社区修正流程（类似 Circuit Breaker）；  
- **多签治理**：关键资产与关键操作纳入多签或 DAO 多人审批，避免单点滥用。

---

## 🔐 安全与信任模型（Trust-Minimized）

在发行阶段（例如 token 在 `pump.fun` 上线）会做去信任化处理：

- 销毁/移除所有 API keys、部署凭证与私人密钥；  
- 把关键权限迁移至多签或链上治理；  
- 不保留创始人可直接操控的后门。

目标：上线后**无人能单独控制** Jester，保障系统的去中心化与可审计性。

## 💰 代币经济与收益分配（Trader-Friendly）

### 开发者代币（Dev Tokens）
- **100% 锁仓/质押机制**，禁止即时抛售；  
- 无暗箱抽取或隐蔽分配机制。

### 收益模型
- Jester 的收入来自 `pump.fun` 的交易手续费或协议级收益；  
- 不依赖订阅或广告，分配透明：
  - 部分用于系统维护（运维成本）；  
  - 部分作为创始人/贡献者版税（仅报酬，不授予控制权）。

对交易员：资金流公开、可审计，降低利益冲突风险。

## 🧩 自治模式补充说明

- **持续运行**：Jester 在 CI / 自动化环境（如 GitHub Actions、云函数）中持续运作；只要自动化存在，Jester 就在工作；  
- **治理扩展**：可无缝对接 Snapshot、链上 Governor 等机制，以实现 on-chain 快照或提案；  
- **适用场景**：AMA、空投、白名单、流动性激励、活动公告均可通过规则治理处理。

## 🛠 故障与边缘情形（运营 & 风控）

| 场景 | 处理策略 |
|---|---|
| 轻微 Bug | 自动恢复 / 重试 |
| 依赖服务宕机 | 重建节点并重试 |
| 规则冲突引发异常 | 自动回滚或触发社区修正投票 |
| 关键密钥泄露 | 触发紧急应对（多签收回 / 权限重置） |

> 在高风险操作（如大额转账、空投发放）建议增加二次人工/多签确认。
